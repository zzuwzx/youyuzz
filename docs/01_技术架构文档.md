## 二、各端技术选型

### 2.1 PC客户端（电脑端）— 双进程架构

| 层 | 技术 | 版本 | 理由 |
|---|---|---|---|
| 核心服务 | Python | 3.11+ | 爬虫(Playwright)、MTP(IFileOperation COM / win32com Shell)、网盘API、文件操作 |
| Web框架 | FastAPI | 0.110+ | 高性能异步HTTP API，供Electron调用 |
| GUI壳 | Electron | 30+ | 跨平台桌面壳，管理Python子进程 |
| GUI渲染 | React | 18+ | 组件化UI，状态管理清晰 |
| UI框架 | Tailwind CSS | 3+ | 原子化CSS，快速实现极简设计 |
| 打包 | PyInstaller + electron-builder | | Python打包为exe，Electron打包为安装包 |

**进程通信**：Electron启动时自动拉起Python FastAPI服务（localhost:18888），GUI通过HTTP调用后端API。

### 2.2 云端（Cloudflare）

| 组件 | 技术 | 免费额度 | 用途 |
|---|---|---|---|
| 计算 | Cloudflare Workers | 10万次/天 | 授权验证、激活码校验、版本查询、心跳 |
| 数据库 | Cloudflare D1 | 5GB / 500万读行/天 | 用户、兑换码、授权记录 |
| 静态站 | Cloudflare Pages | 无限 | 管理后台（生成兑换码、用户管理） |
| 存储 | Cloudflare R2 | 10GB | VIP进阶代码托管、大气层固件缓存 |
| DNS/CDN | Cloudflare CDN | 免费 | GitHub Releases加速、全球加速 |

### 2.3 Switch插件端

| 组件 | 技术 | 理由 |
|---|---|---|
| 开发框架 | devkitPro + libnx | Switch homebrew 唯一成熟生态 |
| GUI | SDL2 | 轻量跨平台图形库 |
| 网络 | libcurl | HTTP请求、网盘API调用 |
| 安装协议 | DBI MTP | 与PC端一致的安装方式 |

### 2.4 NAS

- 硬件：绿联DX4800
- 协议：SMB/CIFS 文件共享
- 脚本：Shell脚本 + crontab 定时任务

## 三、技术决策记录 (ADR)

### ADR-001：Python + Electron 双进程 vs 纯Python GUI

**决策**：采用双进程架构。

**理由**：
- Python 在爬虫、MTP、网盘API方面生态远优于 Node.js
- Electron 的 GUI 开发效率和美观度远优于 PyQt/Tkinter
- 双进程解耦后各自独立开发、测试、打包，互不阻塞
- 维护成本可控（两个进程通信只走 HTTP，接口固定）

**代价**：启动时需要拉起两个进程，首次启动慢 ~2秒。

### ADR-002：MTP 传输方案 —— 三层渐进策略

**决策**：采用三层渐进方案，优先使用 IFileOperation COM 接口。

**方案对比**：

| 方案 | 可靠性 | 进度回调 | 打包依赖 | 状态 |
|---|---|---|---|---|
| IFileOperation COM (ctypes vtable) | ✅ Windows原生MTP栈 | ✅ 原生 Advise 回调 | ✅ 零额外依赖 | 🎯 Phase 1 开发 |
| win32com Shell CopyHere | ✅ 已验证可用 | ⚠️ 窗口检测+阻尼模拟 | ✅ pywin32 | ✅ 已有代码复用 |
| PyMTP (libmtp) | ⚠️ 兼容性不稳定 | ✅ 逐块回调 | ❌ 需libmtp DLL | ❌ 弃用 |

**理由**：
- 现有 SquidInstallerGUI.py 已通过 win32com Shell.Application + CopyHere 验证了 Windows Shell MTP 栈的可靠性
- IFileOperation 走同一 MTP 栈，但额外提供 Advise(IOperationsProgressDialog) 原生进度回调，且无需 libmtp 等系统级 DLL
- PyMTP 方案被否决：需要额外打包 libmtp DLL，Windows 兼容性不如原生 Shell COM
- IFileOperation COM 类在本系统已验证可通过 CoCreateInstance 创建（CLSID: {3AD05575-...}）
- 类型绑定通过纯 ctypes vtable 实现，不依赖 comtypes 类型库

**实施路径**：
1. Phase 1：先复用现有 CopyHere(1556) + 阻尼进度模拟 → 快速上线
2. Phase 1 后期：手写 ctypes IFileOperation vtable + IFileOperationProgressSink → 真实进度回调
3. 封装统一 MTPTransfer 抽象层，两种后端可互换

### ADR-003：Cloudflare Workers + D1 vs 传统VPS

**决策**：Cloudflare Workers + D1。

**理由**：
- 完全免费（在项目用户量 < 5000 阶段）
- 全球边缘加速，延迟低
- 无需运维服务器
- D1 支持 SQL，迁移成本低
